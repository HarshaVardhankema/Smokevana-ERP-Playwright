1. System Overview

What is SmokeVana ERP?

SmokeVana ERP is a comprehensive Enterprise Resource Planning system built on Laravel framework. It's designed to manage all aspects of a business including:

  • Product catalog and inventory management
  • Sales and Point of Sale (POS) operations
  • Purchase and procurement
  • Customer relationship management (CRM)
  • E-commerce (B2B and B2C)
  • Accounting and financial management
  • Multi-location business support
  • WooCommerce integration
  • Restaurant management

Key Features:

  • ✅ Multi-business and multi-location support
  • ✅ Real-time inventory tracking
  • ✅ Advanced product variations and pricing
  • ✅ Multiple payment gateway integrations
  • ✅ Comprehensive reporting system
  • ✅ RESTful API for mobile and third-party integrations
  • ✅ Multi-language support (15+ languages)
  • ✅ Role-based access control

4. Module 1: Product Management

Overview

The Product Management module handles all product-related operations including product creation, variations, pricing, inventory tracking, and categorization.

Key Features:

  • Single and Variable product types
  • Product variations (size, color, etc.)
  • Multi-location inventory tracking
  • Product categories and sub-categories
  • Brand management
  • Barcode generation and management
  • Product gallery images
  • Group pricing (different prices for different customer groups)
  • Product order limits
  • Warranty management

Main Controller:

app/Http/Controllers/ProductController.php

Key Models:

  • Product - Main product model
  • ProductVariation - Product variations
  • Variation - Variation details (SKU, price, stock)
  • Category - Product categories
  • Brands - Product brands
  • Unit - Measurement units
  • VariationLocationDetails - Stock per location
  • ProductLocation - Product-location mapping
  • VariationGroupPrice - Group-based pricing

Important Routes:

GET  /products                    # List all products
GET  /products/create             # Create product form
POST /products                    # Store new product
GET  /products/{id}/edit          # Edit product form
PUT  /products/{id}               # Update product
DELETE /products/{id}             # Delete product
GET  /products/view/{id}          # View product details
POST /products/bulk-edit          # Bulk edit products
POST /products/get-variations/{productId}  # Get product variations

Key Database Tables:

  • products - Main product table
  • product_variations - Product variation templates
  • variations - Actual variation instances
  • categories - Product categories
  • brands - Product brands
  • variation_location_details - Stock per location
  • product_locations - Product-location relationships
  • variation_group_prices - Group pricing
  • product_gallery_images - Product images
⚠️ Important Notes:
  • Products can be of type 'single' or 'variable'
  • Variable products must have variations
  • Stock is tracked at variation level, not product level
  • Each variation has its own SKU and barcode
  • Products can be assigned to multiple locations

7. Module 4: Inventory Management

Overview

The Inventory Management module handles stock tracking, adjustments, transfers, and alerts across multiple locations.

Key Features:

  • Multi-location inventory tracking
  • Stock adjustments
  • Stock transfers between locations
  • Stock alerts (low stock notifications)
  • Stock history tracking
  • Real-time stock updates

Main Controllers:

  • StockAdjustmentController.php - Stock adjustments
  • StockTransferController.php - Stock transfers
  • StockAlertController.php - Stock alerts

Key Models:

  • VariationLocationDetails - Stock per location per variation
  • StockAdjustmentLine - Stock adjustment records

Stock Update Flow:

  1. Sale/Purchase transaction created
  2. TransactionUtil updates stock automatically
  3. VariationLocationDetails table updated
  4. Stock alerts checked if enabled
💡 Stock Calculation:
Stock is tracked at the variation_location_details table level. Each variation has stock per location. When a sale occurs, stock is automatically deducted from the appropriate location.

11. Module 8: Payment Processing

Overview

The Payment Processing module handles multiple payment gateways and payment methods for transactions.

Supported Payment Gateways:

  • Authorize.Net - Credit card processing
  • NMI (Network Merchants Inc) - Payment gateway
  • Stripe - Online payments
  • Razorpay - Payment gateway (India)
  • PayPal - PayPal integration
  • Paystack - Payment gateway (Africa)
  • Pesapal - Payment gateway (Africa)
  • MyFatoorah - Payment gateway (Middle East)

Payment Methods:

  • cash - Cash payment
  • card - Card payment
  • cheque - Cheque payment
  • bank_transfer - Bank transfer
  • credit_card - Credit card
  • other - Other payment methods

Main Controllers:

  • TransactionPaymentController.php - Payment management
  • ECOM/PaymentOrderController.php - E-commerce payments
  • B2CECOM/B2CPaymentOrderController.php - B2C payments

Payment Flow:

  1. Customer initiates payment
  2. Payment gateway selected based on configuration
  3. Payment token/nonce generated (for card payments)
  4. Payment processed through gateway
  5. TransactionPayment record created
  6. Transaction payment status updated
  7. Account transactions created (if accounting enabled)
💡 Gateway Configuration:
The payment gateway is configured in config/app.php via gatewayType environment variable. Set USE_GATEWAY in .env file.

Key Methods:

  • payByAuthorize() - Process Authorize.Net payment
  • doSale() - Process NMI payment
  • _doRequest() - Execute payment request

17. Module 14: Multi-Location & Multi-Business

Overview

The system supports multiple businesses and multiple locations per business, enabling enterprise-level operations.

Key Features:

  • Multi-business support
  • Multi-location inventory
  • Location-specific settings
  • Location-based pricing
  • Location-based taxes
  • Stock transfers between locations

Key Models:

  • Business - Business entity
  • BusinessLocation - Physical locations
  • VariationLocationDetails - Stock per location
  • LocationTaxType - Location-specific taxes

Business Structure:

Business (1)
  └── BusinessLocation (N)
        └── Products (N)
        └── Transactions (N)
        └── Users (N)
💡 Multi-Tenancy:
Each business is isolated. All queries should filter by business_id. Locations belong to a business and can have location-specific settings.

⚠️ Important:
Never modify database structure directly. Always use migrations. Test migrations on development environment first.

20. Deployment & Configuration

Environment Configuration

Key environment variables in .env:

Variable Description
APP_NAME Application name
APP_ENV Environment (local, production)
APP_DEBUG Debug mode
DB_CONNECTION Database connection
DB_DATABASE Database name
USE_GATEWAY Payment gateway type
GOOGLE_ADDRESS_API_KEY Google Maps API key

Installation Steps

  1. Clone the repository
  2. Run composer install
  3. Copy .env.example to .env
  4. Generate application key: php artisan key:generate
  5. Configure database in .env
  6. Run migrations: php artisan migrate
  7. Run seeders (if needed): php artisan db:seed
  8. Set up Passport: php artisan passport:install
  9. Set storage link: php artisan storage:link
  10. Configure web server (Apache/Nginx)

Queue Configuration

The system uses Laravel queues for background jobs. Configure queue driver in .env:

  • QUEUE_CONNECTION=database (default)
  • Run queue worker: php artisan queue:work

Cron Jobs

Add to crontab for scheduled tasks:

* * * * * cd /path-to-project && php artisan schedule:run >> /dev/null 2>&1
✅ Best Practices:
  • Always use environment variables for sensitive data
  • Keep .env out of version control
  • Use migrations for all database changes
  • Test on development before deploying to production
  • Keep dependencies updated
  • Enable queue workers for background jobs
